Sequential Workflow

Dynamic Workflow依赖于一个详细的指令提示来强制LLM按顺序运行步骤。这并不完全可靠，因为LLM可能会决定跳过一个步骤，以错误的顺序运行，或者“卡住”，使过程不可预测。

当需要任务以保证的特定顺序发生时，可以使用Sequential Worfolw，按照您列出的顺序运行每个子agent。

Parallel Workflow

Sequential Workflow的每一步都必须等待前一步完成。如果存在几个互不依赖的任务，按顺序运行它们将是缓慢和低效的，造成每个任务不必要地等待的瓶颈。我们可以使用Parallel Workflow来并发地调用Agent。

下面给出一个示例：

Example: Parallel Multi-Topic Research

Tech Researcher - 调研AI/ML新闻

Health Researcher - 调研最近的医学新闻

Finance Researcher - 调研金融和金融科技新闻

Aggregator Agent - 将所有调研结果合并为一个摘要

最后，我们创建一个ParallelAgent统一并发调度三种研究主题的Agent，再将ParallelAgent和用于总结的Agent嵌套在一个SequentialAgent中。

Loop Workflow

Sequential Workflow和Parallel Workflow都是从头到尾运行的。

这种one-shot方法不适用于需要改进和质量控制的任务。如果初始输出不好，没有办法审查并要求重新生成。

此时可以使用Loop Workflow，重复运行一组Agent，直到满足特定条件或达到最大迭代次数，允许系统不断改进自己的工作。

3 Agent Tool

每个实际场景都有通用工具无法处理的独特需求，自定义工具允许我们实现特定的业务逻辑、连接到系统并解决特定于领域的问题。

Google ADK提供了多种自定义工具类型来处理这些场景，包括Custom tools（自定义工具） 和 Built-in tools（内置工具）。

1）Custom tool：

Function Tools：将任意Python函数转变为智能体工具，例如： get_fee_for_payment_method, get_exchange_rate

Long Running Function Tools：智能体可启动任务后继续其他工作，无需等待完成。

Agent Tools：将其他智能体作为工具调用，例如：AgentTool(agent=calculation_agent)

MCP Tools：连接任意兼容MCP协议的服务，例如：Filesystem access, Google Maps, databases

OpenAPI Tools：仅需提供API规范即可获得现成工具

2）Bulti-in tools：

Gemini Tools：利用 Gemini 能力的工具，例如：google_search, BuiltInCodeExecutor

Google Cloud Tools：用于 Google Cloud 服务和企业集成的工具，例如：BigQueryToolset, SpannerToolset, APIHubToolset

自定义工具

任何Python函数都可以成为代理工具，将函数添加到代理的tools=[]列表中，ADK会自动处理其余操作。

该函数最好遵守下面的规则：

  1. 返回 {"status": "success", "data": ...} 或 {"status": "error", "error_message": ...} 格式的字典。
  2. 在Returns中说明字典中每个key-value对的具体类型。
  3. 在Args中说明工具的适用场景及使用方法。

Callback机制

Plugin是一个自定义的代码模块，可以在Agent生命周期的各个阶段依赖Callback机制自动运行。

如下是常见的callback：

  • before_agent_callback → 在Agent开始处理请求之前运行
  • after_agent_callback → 在代理完成请求任务后运行
  • before_tool_callback / after_tool_callback → 在工具被调用前/后
  • before_model_callback / after_model_callback → 在LLM API被调用前/后
  • on_model_error_callback → 当错误发生

只需要在Runner上注册一个Plugin，即可根据定义自动应用于Agent、工具调用和LLM请求。
